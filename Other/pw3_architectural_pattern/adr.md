# Запись принятого решения № 1: Выбор архитектурного паттерна для системы MetalDefectTracker (MDT)

**Дата:** 05.03.2026

---

## Контекст

Разрабатывается система **MetalDefectTracker (MDT)** - веб-ориентированная система автоматизированного контроля качества поверхности листового металла для металлопрокатного предприятия. Система предназначена для развёртывания в **локальной сети предприятия (on-premise)**.

### Ключевые характеристики системы

- **Стек разработки:** Python (Django) + React, PostgreSQL, MinIO.
- **Пользователи:** три роли - оператор ОТК, технолог, администратор.
- **Домены системы:** аутентификация и управление доступом, приём изображений с камер, автоматическая детекция дефектов (ML), верификация и разметка оператором, аналитика и история, администрирование (камеры, модель, пользователи).
- **Особый компонент:** ML-модель детекции дефектов, которая должна обновляться независимо от остальной системы, поддерживать дообучение и откат к предыдущей версии без остановки основного процесса контроля.

### Ограничения и требования, определяющие выбор

| Требование                                                                                                                                   | Источник                |
| ------------------------------------------------------------------------------------------------------------------------------------------------------ | ------------------------------- |
| Ролевое разграничение доступа (оператор / технолог / администратор)                            | NFRQ-1.1                        |
| Защищённая передача данных внутри сети предприятия (TLS)                                                  | NFRQ-1.3                        |
| Веб-интерфейс, доступный в современных браузерах (Chromium, Gecko)                                           | NFRQ-2.1                        |
| Интуитивно понятный интерфейс для верификации дефектов без длительного обучения | NFRQ-2.2                        |
| Управляемое обновление и откат ML-модели без деградации качества                                 | NFRQ-3.1                        |
| On-premise развёртывание в локальной сети предприятия                                                            | Контекст проекта |
| Малая команда разработки и поддержки                                                                                   | Контекст проекта |

## Решение

Принято решение использовать **модульный монолит с вынесенным ML-сервисом**.

### Описание архитектуры

Система разделена на **два архитектурных кванта** - наименьших независимо деплоящихся единицы с высокой функциональной связностью:

```
┌──────────────────────────────────────────────────────────────────────┐
│  Квант 1: Web Application                                            │
│                                                                      │
│  ┌──────────────┐   REST API    ┌─────────────────────────────────┐  │
│  │  React SPA   │ ◄───────────► │      Django - Модульный монолит │  │
│  │  (Frontend)  │               │                                 │  │
│  └──────────────┘               │  ┌────────────┐ ┌───────────┐   │  │
│                                 │  │  accounts  │ │ receiver  │   │  │
│                                 │  ├────────────┤ ├───────────┤   │  │
│                                 │  │ detection  │ │verificat. │   │  │
│                                 │  ├────────────┤ ├───────────┤   │  │
│                                 │  │ analytics  │ │  admin    │   │  │
│                                 │  └────────────┘ └───────────┘   │  │
│                                 └──────────┬──────────────┬───────┘  │
│                                            │              │          │
│  ┌──────────────────────────────┐  ┌───────▼──────┐ ┌────▼──────┐    │
│  │  Celery Workers              │  │  PostgreSQL  │ │   MinIO   │    │
│  │  (тот же образ, что Django)  │  │  (данные)    │ │  (файлы)  │    │
│  └──────────────┬───────────────┘  └──────────────┘ └───────────┘    │
│                 │  задачи детекции                                   │
│  ┌──────────────▼───────────────┐                                    │
│  │  Redis (broker очереди задач)│                                    │
│  └──────────────────────────────┘                                    │
└──────────────────────────────────────────────────────────────────────┘
           │  HTTP (multipart, байты изображения)  │
           ▼                                       ▼
┌──────────────────────────────────────────────────────────────────────┐
│  Квант 2: ML Inference Service                                       │
│                                                                      │
│  FastAPI/Flask  +  ONNX/PyTorch model  (N реплик в K8s)              │
│                                                                      │
│  - Принимает изображение в теле HTTP-запроса (multipart/form-data)   │
│  - Возвращает JSON со списком дефектов (тип, bbox, confidence)       │
│  - Деплоится независимо через Kubernetes rolling update              │
│  - K8s Service балансирует нагрузку между репликами                  │
│  - Поддерживает откат к предыдущей версии (rollback)                 │
└──────────────────────────────────────────────────────────────────────┘
```

### Модули монолита (Django apps)

| App                | Зона ответственности                                                                 | Требования   |
| ------------------ | ------------------------------------------------------------------------------------------------------- | ---------------------- |
| `accounts`       | Аутентификация, пользователи, роли, участки                        | FRQ-1, FRQ-2           |
| `receiver`       | Приём изображений от камер, создание записи контроля       | FRQ-3                  |
| `detection`      | HTTP-клиент к ML-сервису, сохранение результатов, статусы     | FRQ-4, FRQ-5           |
| `verification`   | Верификация оператором, ручная разметка, аудит действий | FRQ-6, FRQ-7           |
| `analytics`      | Статистика операторов и технологов, история, фильтрация | FRQ-8, FRQ-9           |
| `administration` | Настройка камер, справочник дефектов, версии модели         | FRQ-10, FRQ-11, FRQ-12 |

Взаимодействие между модулями осуществляется **только через публичный Python API** (импорт сервисных функций), но не через прямые SQL-запросы к чужим таблицам - это обеспечивает границы модуля и сопровождаемость.

### Поток обработки изображения (receiver → detection → verification)

Поступление изображения с камеры обрабатывается **асинхронно** во избежание блокировки входящего потока на время ML-инференса:

```
Camera
  │
  └─► POST /api/receiver/images/
          │
          ├─ receiver: сохраняет файл в MinIO
          ├─ receiver: создаёт InspectionRecord (status=queued)
          ├─ receiver: возвращает 202 Accepted  ← камера немедленно свободна
          │
          └─ receiver: помещает задачу detection_task в Redis
                                    │
                           [Redis - очередь]
                                    │
                           Celery Worker (detection module):
                            ├─ читает байты изображения из MinIO
                            ├─ POST к ML-сервису (multipart)
                            ├─ получает [{type, bbox, confidence}, ...]
                            ├─ сохраняет DetectionResult записи в Postgres
                            └─ обновляет InspectionRecord (status=pending)
                                               │
                           React SPA: polling /api/images/ или WebSocket
                                               │
                           verification: оператор видит изображение + bboxes
                            ├─ подтверждает / отклоняет найденные дефекты
                            ├─ удаляет ложные срабатывания
                            ├─ добавляет пропущенные вручную (bbox + тип)
                            └─ устанавливает итоговый статус: accepted / rejected
```

При нескольких активных Celery Workers K8s Service автоматически балансирует их HTTP-запросы между репликами ML-сервиса.

### Инфраструктура

- **React SPA** - отдельный деплоймент статики, взаимодействует с Django исключительно через REST API. Не использует Django Templates. Это обусловлено требованием к богатому интерактивному интерфейсу верификации (рисование bbox на изображении) - NFRQ-2.2.
- **PostgreSQL** - хранит все реляционные данные. Изображения не хранятся в БД.
- **MinIO** - S3-совместимое объектное хранилище, развёртывается on-premise. Django взаимодействует с ним через `django-storages`. Nginx отдаёт файлы напрямую.
- **Nginx** - reverse proxy: маршрутизирует `/api/` → Django, `/` → React SPA, `/media/` → MinIO/файловая система.
- **Celery + Redis** - инфраструктурный инструмент асинхронного выполнения задач внутри кванта 1. Celery Workers деплоятся из того же Docker-образа, что и Django, используют те же модели и ту же БД. Redis выступает брокером очереди задач. Применяется для асинхронной отправки изображений в ML-сервис, чтобы не блокировать входящий поток с камеры.
- **Kubernetes** - управляет деплоем кванта 2 (ML-сервиса): rolling update обеспечивает замену версии модели без остановки сервиса, rollback - возврат к предыдущей версии при деградации качества (NFRQ-3.1). K8s Service (ClusterIP) балансирует нагрузку между репликами ML-Pod.

## Последствия

### Преимущества

- **Простота операционного сопровождения.** Один основной деплоймент (Django) покрывает ~95% логики. Отсутствует необходимость в service mesh, distributed tracing и централизованном логировании между сервисами - критично для on-premise с малой командой.
- **Надёжный RBAC.** Ролевое разграничение реализуется через Django middleware и DRF permissions в единой кодовой базе - прозрачно, тестируемо, без межсервисных токенов (NFRQ-1.1).
- **Целостность данных.** Все операционные данные (пользователи, изображения, дефекты, действия операторов, статистика) находятся в одной БД - нет проблемы распределённых транзакций при аналитических запросах агрегации по всем операторам (FRQ-9.3, FRQ-9.6).
- **Независимый цикл обновления ML.** Kubernetes rolling update для кванта 2 позволяет обновлять модель без остановки системы контроля. Единственная межквантовая точка - HTTP POST с байтами изображения и JSON-ответ. Несколько реплик ML-сервиса и несколько Celery Workers обеспечивают параллельную обработку потока изображений (NFRQ-3.1, FRQ-12).
- **Асинхронность не меняет архитектурный паттерн.** Введение Celery + Redis - инфраструктурное решение внутри кванта 1, а не переход к событийно-ориентированной архитектуре (EDA). В EDA брокер является основным механизмом связи между независимыми компонентами, которые не знают друг о друге. Celery Workers напротив разделяют кодовую базу, модели данных и БД с Django - они неотделимы при деплое и не являются самостоятельными компонентами.
- **Путь к масштабированию.** Чётко определённые модули могут быть вынесены в отдельные сервисы по мере роста системы без переписывания бизнес-логики.
- **Богатый UI без ограничений.** Отдельный React SPA не ограничен возможностями Django Templates и позволяет реализовать canvas-разметку дефектов, интерактивную фильтрацию и аналитические дашборды (NFRQ-2.1, NFRQ-2.2).

### Недостатки и риски

- **Монолит требует дисциплины.** При росте команды без соблюдения границ модулей монолит деградирует в «большой комок грязи». Необходимо code review с проверкой межмодульных зависимостей.
- **Единая точка отказа.** Падение Django-процесса останавливает весь функционал (кроме ML-сервиса). Митигируется несколькими репликами в K8s или процесс-менеджером (Gunicorn workers).
- **Общая БД как узкое место.** При высокой нагрузке (много камер, частые запросы) PostgreSQL может стать бутылочным горлышком. Для текущего масштаба (~1 предприятие) некритично; при масштабировании решается через read-реплики.
- **Celery Workers как дополнительный процесс.** Необходимо следить за здоровьем воркеров отдельно от Django-процесса. Митигируется мониторингом через Flower (веб-интерфейс Celery) или стандартными K8s liveness-пробами.

## Альтернативы

### Альтернатива 1: Слоистая архитектура (Layered Monolith)

Код организован горизонтально: слой представления → слой сервисов → слой репозиториев.

**Преимущества:**

- Проще в освоении новым членом команды.
- Меньше структурных решений на старте.

**Недостатки по сравнению с выбранным решением:**

- Слои режут код горизонтально, тогда как домены MDT (верификация, аналитика, администрирование) - это вертикальные срезы. Логика одного домена оказывается размазана по всем слоям одновременно.
- При добавлении нового функционала (например, новый тип дефекта) необходимо вносить изменения в каждый слой, а не в один модуль.
- Труднее разграничить ответственность и назначить владельца для каждого домена.

**Причина отказа:** слоистая архитектура не даёт структурной изоляции доменов, которая необходима для сопровождаемости системы с 6 различными функциональными областями (NFRQ-3).

---

### Альтернатива 2: Архитектура на основе сервисов (Service-based Architecture)

4–6 крупных независимо деплоящихся сервисов по доменам: Users Service, Images Service, Analytics Service и т.д., с общей или частично разделённой БД.

**Преимущества:**

- Независимое масштабирование тяжёлых доменов (например, Analytics).
- Независимый деплой каждого домена.

**Недостатки по сравнению с выбранным решением:**

- Значительно усложняет операционную инфраструктуру: несколько CI/CD пайплайнов, сервисные учётные данные, cross-service аутентификация - overhead без реальной пользы при on-premise развёртывании малой командой.
- Аналитические запросы, агрегирующие данные по пользователям, изображениям и дефектам одновременно (FRQ-9.3, FRQ-9.6), требуют либо общей БД (что нивелирует независимость сервисов), либо сложных механизмов синхронизации данных.
- Домены MDT сильно связаны по данным: верификация ссылается на результаты детекции, аналитика агрегирует данные верификаций и пользователей.

**Причина отказа:** операционная сложность неоправдана при текущем масштабе и условиях развёртывания. Модульный монолит обеспечивает те же границы доменов с существенно меньшими издержками на сопровождение.

---

### Альтернатива 3: Микросервисы

Каждый функциональный модуль - отдельный независимый сервис со своей БД.

**Преимущества:**

- Максимальная независимость деплоя и масштабирования.
- Технологическая гетерогенность (разные сервисы на разных стеках).

**Недостатки по сравнению с выбранным решением:**

- Требует service mesh (Istio/Linkerd), distributed tracing (Jaeger), centralized logging (ELK), API gateway - значительный инфраструктурный overhead.
- Распределённые транзакции: подтверждение дефекта оператором затрагивает данные верификации, изображения, статистики и аудита одновременно. В микросервисах это требует Saga-паттерна или событийной согласованности.
- Несоразмерная сложность для системы, которая разрабатывается и поддерживается малой командой на одном предприятии.

**Причина отказа:** микросервисная архитектура решает проблемы масштаба и независимости команд, которых в данном проекте нет. Введение этого паттерна создаст существенные операционные риски без архитектурной выгоды.
